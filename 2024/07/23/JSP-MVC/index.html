# JSP-MVC

# 1.JSP

# 1.1.HTML 和 Servelet 的弊端

​ HTML：

​ 优点：方便页面的布局及美化

​ 缺点：不能制作动态内容

​ Servlet：

​ 优点：可以制作动态内容

缺点：不方便写CSS和JS代码

二者的优缺点刚好互补，两者综合起来就刚刚好是 JSP 的设计思想

# 1.2.JSP 简介

​ 以 Java 作为脚本语言，部署在网络服务器上，可以根据客户端的请求内容动态生成 HTML、XML 或其他格式文档的 Web 网页，然后返回给请求者。另外，JSP 的标签库 JSTL 提供了类似于 Java 中的 if、switch、for 等功能，开发者可以更方便地实现条件渲染和列表渲染等操作，而且也不受跨平台的限制。

**Tips：**JSP 编译器可以将 JSP 文件编译成用 Java 代码写的 Servlet，然后 Servlet 再编译生成字节码文件。JSP 本质上就是一个 Servlet，属于动态技术。

# 1.3.JSP 语法

# 1.3.1. 输出脚本

​ 作用： 直接输出变量值或对表达式进行计算

<%= 变量名称或表达式%>  // 相当于Servlet的out.print()

# 1.3.2. 声明

**作用：** 声明全局变量或方法

<%! 全局变量或方法 %>

# 1.3.3. 插入代码片段

**作用：** 插入Java代码

<% Java代码 %>

# 1.4.JSP 九大内置对象

​ PageContext: 页面域

​ Request： HttpServletRequest (请求域)

​ Session： 会话域

​ Application： 上下文域

以上为四大域对象， 从大到小依次为 Application > Session > Request > PageContext

相关的方法：

作用域 Java 代码 EL 的写法 范围大小
页面域 <% = pageContext.getAttribute ("键名") %> $ 1
请求域 <% = request.getAttribute ("键名") %> $ 2
会话域 <% = session.getAttribute ("键名") %> $ 3
上下文域 <% = application.getAttribute ("键名") %> $ 4
自动查找 <% = pageContext.findAttribute ("键名") %>

​ Response： HttpServletResponse

​ Out

​ Config： ServletConfig

​ Page: 当前页面的引用，就是 this

​ Exception： 封装了出现的异常， 只有在错误页面的中才会存在

# 1.4.1PageContext 域

​ PageContext 域是 servlet 中 4 个域对象中最后一个了，而且大小也是最小的，只在当前页面下生效，且在不同的页面下不能实现页面共享（即使是一次请求）

​ 有关的方法：

方法 说明
void setAttribute(String key, Object value) 向页面域中添加键和值
Object getAttribute(String key) 从页面域中得到值
void removeAttribute(String key) 删除页面域中键值对
Object findAttribute(String key) 自动从四个作用域中去查某个键，从小到大的范围来查找，如果找到就停止。如果没有找到，返回 null，该方法是页面域独有的

# 1.4.2Exception 对象

​ Exception 属于 JSP 的九大内置对象，但是只有错误页面才会生成该对象，普通的 JSP 页面不会存在 Exception 对象

​ warning：

​ (1) 当前页面只有设置了 isErrorPage = true 时才可以使用 exception 内置对象

​ (2) 常规页面只有八大内置对象，只有错误页面才有九大内置对象

# 2.EL 表达式

​ 语法

${ 变量名或表达式}

# 2.1. 食用说明

​ 变量名： 变量名即可以是诸如 x, y, z 这样的普通变量，也可以是 Object、List、Map、数组或者 JavaBean 等

Tips: 键名中如果有特殊字符时，可以使用双引号或者单引号将其引起来。 e.g. $

​ 表达式： EL 表达式中的表达式可以包含常规的算术表达式（ + 、 - 、* 、/ 、%）、比较表达式（ == 、 != 、<、<=、> 、>=）、逻辑表达式（&& 、|| 、！）、三元运算以及判空表达式（${ empty 变量名}）

# 3.1 相关标签

**<c:if test="条件判断表达式"> ** 条件判断

<c:choose> 多分支判断

类似于 switch，层级结构为：

<c:choose>
    <c:when test = "条件表达式">
   	 	xxxxxxxx
    </c:when>
    //默认条件，相当于default
    <c:otherwise>
        xxxxxxx
    </c:otherwise>
</c:choose

<c:forEach> 循环渲染

类似于 for 语句，相关的参数有

属性名 EL 支持 类型
items true 数组或集合
var false String
varStatus false String
begin true int
end true int
step true int

其中 varStatus 还有一些子属性（虽然也不是很常用）

属性 数据类型 含义
index int 当前元素的索引号，从 0 开始
count int 遍历到当前位置，一共遍历了多少个元素（默认从 1 开始）
first boolean 如果当前遍历的是第 1 个元素，则返回 true
last boolean 如果当前遍历的是最后 1 个元素，则返回 true

# 4. 三层架构 And MVC

# 4.1. 三层架构

三层架构（Three-Tier Architecture）和 MVC（Model-View-Controller）是两种常见的软件设计模式，它们用于构建可维护、可扩展和模块化的应用程序。

三层架构是一种分层的客户端 - 服务器架构，它将应用程序分为三个逻辑层次：

  1. 表示层（Presentation Layer）：这一层处理用户界面和用户交互。它负责接收用户的输入，调用业务逻辑层来处理数据，并显示信息给用户。

  2. 业务逻辑层（Business Logic Layer）：也称为中间层或服务层，它包含应用程序的业务规则和逻辑。这一层处理数据的加工和计算，如数据库操作、数值计算等。

  3. 数据访问层（Data Access Layer）：这一层负责数据持久化，包括数据的读取、写入和更新。它通常与数据库交互，执行 CRUD（创建、读取、更新、删除）操作。

三层架构的优点包括：

  • 分离关注点：每一层都有其特定的职责，这有助于分离关注点，使得开发和维护更加容易。
  • 可维护性：每一层可以独立开发和测试，提高了代码的可维护性。
  • 可扩展性：可以根据需要扩展任何一层，而不会影响到其他层。

# 4.2.MVC

MVC 是一种设计模式，用于实现用户界面的分离。它将应用程序分为三个主要组件：

  1. 模型（Model）：代表应用程序的数据和业务逻辑。模型负责处理数据和业务规则。

  2. 视图（View）：代表用户界面。视图负责显示数据（模型）和接收用户输入。

  3. 控制器（Controller）：作为模型和视图之间的中介。控制器接收用户的输入，调用模型处理数据，然后选择合适的视图显示结果。

MVC 的优点包括：

  • 分离关注点：MVC 通过将数据（模型）、用户界面（视图）和业务逻辑（控制器）分离，使得开发和维护更加清晰。
  • 可维护性：由于关注点的分离，MVC 使得代码更加模块化，易于维护和测试。
  • 可重用性：模型和视图可以独立于控制器重用，提高了代码的重用性。

# 4.3. 三层架构与 MVC 的关系

三层架构和 MVC 可以结合使用。在这种结合中：

  • 表示层通常包含 MVC 的视图和控制器。视图负责显示数据，控制器处理用户输入和调用业务逻辑层。
  • 业务逻辑层对应于 MVC 的模型，处理应用程序的核心逻辑。
  • 数据访问层通常不直接包含在 MVC 中，但它为模型提供数据持久化的支持。

通过将三层架构与 MVC 结合，可以构建一个结构清晰、易于维护和扩展的应用程序。
